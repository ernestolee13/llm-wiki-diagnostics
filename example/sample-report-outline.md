# 샘플 리포트 윤곽

이 문서는 최종 보고서가 어떤 느낌이어야 하는지 보여주는 익명화된 참고 예시다. 실제 실행 시에는 사용자의 위키 구조와 수집 가능한 지표에 맞게 항목을 줄이거나 바꾼다.

예시의 역할은 `판단 구조`, `용어의 친절함`, `상단 요약과 하단 근거의 배치`, `시각화 밀도`를 보여주는 것이다. 예시 수치, 파일명, 문단 순서, 차트 종류는 고정 템플릿이 아니다.

## 한눈에 보기

이 영역은 숫자만 나열하지 않고, 작은 카드와 미니 차트로 `지금 위키 상태를 어떻게 읽어야 하는지`를 먼저 보여준다. 각 카드는 값, 기준 분모, 신뢰도, 한 줄 해석을 함께 둔다.

상단 1-2 화면은 사용자가 바로 판단할 수 있는 내용에 집중한다. 핵심 지표와 추세, 핵심 분석, 추천 방향을 압축하고, 긴 원자료 표와 후보 목록은 아래로 내린다.

### 위키가 잘 구축되어 있는가?

부분적으로 잘 구축되어 있다. 가이드, 글로벌 인덱스, 프로젝트 하위 인덱스, 운영 로그가 있어 에이전트가 시작할 수 있는 기본 경로는 있다. 다만 일부 문서는 현재 연결 모델에서 들어오는 연결이 약해 검색이나 인덱스에 의존할 가능성이 있다.

추천 차트:

- 문서 수와 운영 기록의 주간 변화
- 문서 읽기 부담 분포
- 진입점별 추정 토큰 크기
- 연결 점검 필요 문서 비율

### 내가 효율적으로 사용하고 있는가?

회고 추정 기준으로 검색성 요청은 평균 3.1개의 검색/조회 단계를 거치고 4.4개의 문서를 열어본다. 아직 쿼리 로그가 없어 정확한 검색 시간과 토큰은 추정치다. 최근 주간 추세에서 요청당 확인 문서와 소요 시간이 함께 늘면 위키가 무거워지는 신호로 볼 수 있다.

세션을 간접 분석할 때는 전체 대화량만 보지 말고, 위키와 직접 관련된 요청을 따로 좁혀서 해석한다. 가능하면 `전체 요청`, `target 관련 요청`, `조회/조사`, `작성/정비`, `조회+작성 혼합`, `대화/계획`처럼 요청 유형을 나누고, 각 유형별 요청당 비용을 비교한다. 그래야 일반 대화나 다른 프로젝트 작업이 위키 사용 비용을 부풀리는 일을 줄일 수 있다.

추천 차트:

- 요청당 검색/조회 단계 추세
- 요청당 열어본 문서 수 추세
- 검색성 요청의 보통 소요 시간과 오래 걸린 요청 추세
- 검색성 요청 토큰 추세
- target 관련 요청과 전체 요청의 비용 비교
- 요청 유형별 도구 호출, 읽기/쓰기, 시간, 토큰 신호 비교

## HTML 표현 예시

이 섹션은 구현 요구사항이 아니라, 생성된 HTML 리포트가 읽기 좋게 보이도록 돕는 심미적 참고다. 실제 리포트는 사용자의 데이터와 에이전트 환경에 맞게 단순화하거나 바꿀 수 있다.

- 상단은 `한눈에 보기` 카드 4-6개로 시작한다. 예: `위키 규모`, `인덱스와 진입점`, `문서 읽기 부담`, `연결 점검 필요`, `요청당 사용 비용`, `데이터 공백`.
- 각 카드는 큰 숫자 하나만 두지 말고, 작은 보조 문장과 신뢰도 배지를 둔다. 예: `inferred`, `measured`, `unavailable`.
- 추세가 있으면 카드 안에 아주 작은 sparkline이나 mini bar를 넣고, 추세가 없으면 `추세 데이터 없음`을 시각적으로 표시한다.
- 색상은 의미를 가진 3-4개 수준으로 제한한다. 예: 정상/주의/데이터 공백/추정. 장식용 그라데이션보다 정보 구분이 우선이다.
- 긴 표는 하단으로 내리고, 상단에는 denominator bar, horizontal bar, stacked bar, histogram, mini timeline 같은 작은 차트를 둔다.
- 차트 캡션은 항상 `무엇을 봐야 하는지`를 한 문장으로 말한다. 예: `문서 수가 늘어도 요청당 읽는 문서 수가 안정적이면 구조가 아직 버티는 중일 수 있다.`
- 섹션 제목은 기술어보다 질문형으로 둔다. 예: `문서를 찾기 쉬운가?`, `인덱스가 길어지고 있나?`, `자주 쓰는 지식이 약하지 않은가?`
- 카드 안에는 긴 설명을 넣지 않는다. 자세한 정의, 계산 방식, caveat는 카드 아래의 세부 지표 섹션으로 보낸다.
- 모바일에서도 카드가 1열로 자연스럽게 쌓이고, 차트 텍스트가 겹치지 않아야 한다.

## 핵심 분석

상단 핵심 분석은 두 큰 질문을 각각 하나의 묶음으로 두고, 그 안에 3-4개의 하위 질문과 짧은 답을 둔다. 아래 문장은 예시이며, 실제 리포트는 사용자의 지표와 용어로 바꾼다.

### 위키가 잘 구축되어 있는가?

- `운영 구성요소가 갖춰져 있는가?`: 가이드, 스키마, 인덱스, 로그, 세션 근거가 확인되는지 말한다.
- `인덱스와 시작 지점이 탐색을 돕는가?`: 글로벌 인덱스, 하위 인덱스, route entrypoint가 충분하고 너무 크지 않은지 말한다.
- `운영 기록과 신뢰 단서가 남는가?`: 추가/수정/정리 기록, source, updated, status 같은 신뢰 단서가 있는지 말한다.
- `문서 단위와 연결은 문제가 없는가?`: 문서 길이, stub/orphan/dead-end/broken-reference 후보를 검토 후보로 설명한다.

### 내가 효율적으로 사용하고 있는가?

- `요청당 비용은 어떤가?`: 요청당 검색, 조회, 도구 호출, 읽은 객체, 시간, 토큰을 측정 또는 추정한다.
- `시간이 지나며 무거워지는가?`: 요청량뿐 아니라 요청당 시간, 토큰, 도구 호출, 조회 객체 수의 추세를 같이 보여준다.
- `무엇을 자주 쓰는가?`: hot object가 실제 내용인지, 시작 가이드/인덱스/로그/생성물인지 구분한다.
- `어디서 비용이나 실패가 생기는가?`: 요청 유형, 경로, route depth, selected rank, hit/fallback, 측정 공백을 연결해서 설명한다.

이 하위 질문은 아래 세부 차트 제목으로만 흩어지면 안 된다. HTML 상단 `핵심 분석` 영역에서 질문과 짧은 답이 먼저 보여야 하고, 세부 차트는 그 근거를 더 자세히 보여주는 역할을 한다.

같은 질문별 답변 요약은 HTML에만 숨기지 말고 aggregate artifact에도 남기는 편이 좋다. 예를 들어 `diagnostic_question_groups` 안에 질문, 짧은 답, 근거 수치, source, confidence를 넣으면 다음 재생성이나 사용자 피드백 때 어느 단계의 정의를 바꿔야 하는지 추적하기 쉽다.

### 인덱스와 진입점이 적절히 설정되어 있는가?

글로벌 인덱스와 프로젝트 하위 인덱스가 있어 기본 진입점은 확인된다. 여기서 `진입점`은 에이전트가 처음 읽고 다음 문서로 이동하는 시작 페이지나 route를 뜻한다. 가장 큰 인덱스가 약 1,400 토큰으로 추정되므로 아직 즉시 위험한 수준은 아니지만, 문서 수와 조회 비용이 같이 늘면 주제별 하위 인덱스가 더 필요해질 수 있다.

### 문서가 검색하고 읽기 좋은 단위인가?

보통 문서는 약 360 단어이고 긴 문서 기준선은 약 1,850 단어다. 긴 문서가 자주 조회되는 경우에는 섹션 분리, 요약 인덱스, 목적별 문서 분리가 검색 비용을 줄일 수 있다.

### 사용 비용이 시간이 지나며 무거워지고 있는가?

주간 추세 예시에서는 검색/조회 단계와 열어본 문서 수가 함께 증가했다. 이는 단순히 사용량이 늘어난 것일 수도 있지만, 같은 종류의 질문에서 반복된다면 진입점 정리나 검색 경로 개선이 필요하다는 신호다.

시간과 토큰은 단위가 특히 중요하다. 세션 timestamp 기반 시간은 idle gap, 세션 재개, 긴 tool 대기 시간을 포함할 수 있으므로 p50/p90과 outlier caveat를 함께 보여준다. provider usage field는 실제 request log의 token delta가 아닐 수 있으므로 `token signal`, `provider usage`, `estimated text tokens`, `measured request delta`를 섞어 그리지 않는다.

## 추천 방향

1. 구조: 자주 찾는 주제는 하위 인덱스나 허브로 묶고, 연결 보강 후보는 실제 사용 빈도가 있는 것부터 처리한다.
2. 사용 흐름: 검색성 요청과 유지보수 요청을 구분해 요청당 비용을 본다. 총합보다 요청당 검색 단계, 문서 수, 시간, 토큰이 더 직관적이다.
3. 텔레메트리: 다음 주기부터 요청 ID, elapsed time, retrieval route, selected objects, token fields, hit status를 남기면 추정치를 실측으로 바꿀 수 있다.

## 세부 지표와 시각화

아래 영역은 원자료 시각화 중심으로 구성한다.

- 구조 지표: 문서 수, 문서 읽기 부담 분포, 인덱스 크기, 진입점 수
- 연결 지표: 연결 점검 필요 문서, dead-end 후보, broken link 후보
- 연결 우선순위: live content, archive, generated/diagnostic output, template/source example을 구분해 실제 운영에 영향을 줄 후보를 먼저 보여준다
- 운영 지표: 로그 엔트리 수, 주간 추가/수정/정리 추세
- 사용 지표: 요청당 검색 단계, 요청당 열어본 객체, 보통/오래 걸린 요청 시간, 토큰 추정
- 데이터 공백: 쿼리 로그 없음, token telemetry 추정, elapsed time 추정

세부 영역은 더 길어도 된다. 다만 각 차트나 표에는 `무엇을 뜻하는지`, `어떤 분모인지`, `측정/추정/공백 중 무엇인지`, `다음 재수집 때 무엇을 고치면 좋은지`가 짧게 붙어야 한다.

## 재생성 가이드

값이 이상하면 HTML을 직접 고치지 않는다. 구조 분석에서 빠진 로그/인덱스/세션 소스가 있는지 확인하고, 지표 수집 규칙을 수정한 뒤 리포트를 다시 생성한다.
